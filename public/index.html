📐 Persamaan Matematis

1. Arrival Time (Waktu Kedatangan)

Arrivali = Arrivali-1 + IAi

Dimana IAi adalah Interarrival Time transaksi ke-i. Arrival1 = 0.

2. Start Time (Waktu Mulai Verifikasi)

Starti = max(Arrivali, AvailableServerj)

Transaksi mulai diverifikasi pada waktu maksimal antara kedatangan dan waktu server j tersedia.

  • Jika server tersedia lebih awal: mulai = waktu kedatangan
  • Jika server sibuk: mulai = waktu server bebas (harus menunggu)

3. End Time (Waktu Selesai Verifikasi)

Endi = Starti + VTi

Transaksi selesai pada waktu mulai ditambah durasi verifikasi (VT ~ Uniform[2, 6] detik).

4. Waiting Time (Waktu Tunggu)

Waitingi = Starti - Arrivali

Lama transaksi menunggu di antrian sebelum diproses.

  • Waitingi = 0 → transaksi langsung diproses (server tersedia)
  • Waitingi > 0 → transaksi harus menunggu (congestion)

5. Queue Length Dynamics

Queue(t+1) = Queue(t) + Arrival(t) - Departure(t)

Panjang antrian berubah berdasarkan kedatangan (+1) dan keberangkatan (-1) transaksi.

6. Server Utilization

Utilization = Σ VTi / (Nserver × Ttotal) × 100%

Persentase waktu server digunakan dibandingkan total waktu simulasi × jumlah server.

🖥️ Server Allocation Strategy

Earliest Available Server (EAS) + FCFS

  1. Ketika transaksi tiba, sistem mencari server dengan available_until paling awal
  2. Jika server tersedia (available_until ≤ arrival_time), transaksi langsung diproses
  3. Jika semua server sibuk, transaksi masuk antrian dan diproses saat server pertama bebas
  4. Semua server identik dengan kapabilitas yang sama
  5. Scheduling mengikuti prinsip First-Come-First-Served (FCFS)

🎲 Monte Carlo Simulation

Mengapa Monte Carlo?

Dalam sistem nyata, verification time bervariasi karena berbagai faktor:

  • 🌐 Network delay
  • ⚡ Server load variations
  • 🏦 Payment gateway latency

Pendekatan Monte Carlo:

  1. Ambil dataset base (interarrival time tetap)
  2. Randomize verification time setiap iterasi: VT ~ Uniform(2, 6) detik
  3. Jalankan DES untuk setiap set VT yang baru
  4. Ulangi 1000 kali (atau sesuai parameter)
  5. Aggregate hasil untuk mendapatkan distribusi metrik

Dengan 1000 iterasi, kita mendapatkan distribusi probabilistik dari metrik kinerja, bukan hanya satu titik estimasi. Ini memungkinkan analisis confidence interval, worst-case scenario, dan probabilitas kejadian.
